Root Cause Analysis

The data from Sprint 5's bugs points to four primary areas of concern:

1. Communication Gaps and Unclear Requirements

A new category of issue was identified this sprint. The bug #23631 ([FE][Search Result] Redundant X icon) was logged by QA based on the ticket's requirements, but the Product Manager's actual expectation had changed without the ticket being updated. This highlights a critical process issue where development and QA are working off outdated information, leading to wasted effort in logging, investigating, and resolving non-issues.

2. Insufficient Developer Self-Testing

A significant number of bugs (18 out of 36, or 50%) are visual or functional defects that could have been caught by the developer before creating a pull request. This includes UI not matching Figma, components overlapping, and incorrect responsive behavior. This suggests a process gap where developers may not be thoroughly validating their work against the provided designs and requirements.

3. Lack of Proactive Accessibility Implementation

With 10 bugs (28%) directly related to screen readers and VoiceOver, it's clear that accessibility is not being consistently addressed during development. These issues are often found late in the cycle by QA, whereas they should be an integral part of component creation. This indicates a potential knowledge gap or lack of emphasis on accessibility within our team's Definition of Done.

4. Component Complexity and Technical Debt

A few components account for a disproportionately large number of bugs. Specifically, Global Navigation (8 bugs), Secondary Nav (5 bugs), and Search Result (4 bugs) make up 47% of all reported issues. This concentration suggests these components may be overly complex or have accumulated technical debt, making them prone to regressions.

Proposed Action Plan

To address these root causes and improve our quality moving forward, the following actions are recommended:

  1. Improve "Ticket Refinement" with PM Sign-off: If the PM needs to change requirements during development, they must update the ticket first. Developers should always verify the latest version of the ticket before starting or continuing development. This ensures requirements are clear, stable, and prevents misalignment.
  2. Impove existing PR Checklist: Before any Pull Request can be merged, developers must complete a mandatory checklist. This will include items like "Verified UI against Figma on all specified viewports", "Tested all interactive states", and "Manually tested core functionality".
  3. Integrate Automated Accessibility Testing: Mandate Pre-PR Accessibility Audits with Browser Tools: Before a Pull Request can be created, developers must run an automated accessibility audit using a browser extension like Axe DevTools. A screenshot showing zero "Critical" violations must be included in the PR description as evidence of the check. This provides immediate feedback and serves as a quality gate before peer review.
